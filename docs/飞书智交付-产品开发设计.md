# Feishu DevFlow Pilot：飞书原生 AI 研发交付中枢

> 目标不是再做一个“会生成代码的聊天框”，而是把飞书中已经存在的需求文档、群聊、任务、审批、多维表格和代码仓库，编排成一条可追踪、可审计、可复用的 AI 研发交付流水线。

## 1. 参赛定位

赛题要求实现一个 AI 驱动的研发流程引擎，覆盖「需求输入 → 方案设计 → 编码 → 测试 → 评审 → 交付」全链路，并具备 Pipeline、Agent 编排、Human-in-the-Loop、API-First 和端到端演示能力。

本项目建议定位为：

**飞书原生 AI 需求到 MR 交付平台。**

一句话描述：

**团队在飞书里提交需求，DevFlow Pilot 自动完成需求澄清、技术方案、代码修改、测试、评审和 MR 创建，并把全流程沉淀回飞书文档、多维表格和消息卡片。**

这个方向的价值点不是“把飞书 API 都接一遍”，而是让飞书成为研发流程的操作入口、组织上下文来源和交付记录系统；DevFlow Engine 负责编排；Agent 负责执行；人类只在关键节点审批。

## 2. 真实问题

研发团队真实卡点通常不是“不会写代码”，而是：

- 需求分散在飞书文档、群聊、会议纪要、任务卡片中，进入研发时上下文不完整。
- PRD 到技术方案、技术方案到代码实现之间依赖人工转述，信息容易损耗。
- 技术负责人需要反复追问影响范围、风险、测试计划和交付状态。
- 代码评审、测试补充、交付总结标准不稳定。
- 管理者很难在一个地方看到每个需求卡在哪个阶段、为什么卡住、谁需要处理。

因此本项目要解决的问题是：

**把飞书协作空间变成 AI Native 的研发流程驾驶舱，让每个需求从进入研发开始就具备结构化状态、产物、审批记录和交付闭环。**

## 3. 产品边界

### 3.1 必须做深的事情

- 以飞书机器人、飞书文档和多维表格作为真实入口，而不是只在 Web 控制台里模拟需求。
- Pipeline 引擎必须能启动、暂停、恢复、终止，并能在 Reject 后携带原因回退重做。
- 每个 Stage 必须有明确输入输出契约，产物可被下一阶段复用，也可展示给人类审批。
- 至少两个 Human-in-the-Loop 检查点必须能真实操作：技术方案审批、代码评审确认。
- 端到端演示必须以项目自身为目标仓库，跑出真实代码 Diff、测试结果和 MR/PR 草稿。

### 3.2 暂时不要做散的事情

- 不要把项目做成“飞书 API 能力陈列柜”。每个飞书能力都必须服务于研发交付闭环。
- 不要一开始实现复杂的企业权限矩阵。MVP 先支持管理员、发起人、审批人三类角色。
- 不要过早做全量代码语义索引。MVP 先支持仓库目录扫描、路径上下文、关键字检索和用户显式指定文件。
- 功能二的网页圈选改代码可以作为后续加分项，不应压过飞书原生交付主线。

## 4. 核心使用场景

### 场景 A：从飞书需求文档一键发起研发流水线

用户在飞书文档里写 PRD，在研发群里 @DevFlow Bot：

```text
@DevFlow 基于这份文档创建新功能研发流程，目标仓库是 Feishu-Pipeline。
```

系统执行：

1. 读取飞书文档正文和标题。
2. 抽取背景、目标、用户故事、验收标准、开放问题。
3. 在多维表格创建一条研发需求记录。
4. 启动 Pipeline Run。
5. 生成技术方案文档。
6. 在飞书群发送可交互消息卡片，请 Tech Lead 审批。

### 场景 B：研发群自然语言派单

用户在研发群里输入：

```text
@DevFlow 为 Pipeline 增加失败阶段自动重试能力，支持最大重试次数配置，并在飞书消息卡片展示当前重试次数。目标仓库是当前平台。
```

Bot 回复动态卡片：

- 需求摘要
- 目标仓库
- 影响模块
- 当前阶段
- 风险等级
- 负责人
- `Approve` / `Reject` 按钮
- 查看方案、查看 Diff、查看测试结果、查看 MR 的入口

### 场景 C：现场命题挑战

评委现场给一个需求，选手直接在飞书群里发起：

```text
@DevFlow 给平台增加一个 Pipeline 模板复制功能，复制后允许修改名称和默认模型。
```

系统跑完整链路：

**需求分析 → 技术方案 → 方案审批 → 代码生成 → 测试生成/执行 → AI Review → 人工确认 → 创建 MR → 飞书交付总结。**

这个场景用于证明平台不是静态 Demo，而是真能驱动自身代码演进。

## 5. 产品形态

建议采用“三端一体”：

### 5.1 飞书机器人：主入口

职责：

- 接收自然语言需求。
- 解析飞书文档链接、仓库、分支、审批人。
- 推送 Pipeline 状态。
- 发起方案审批、代码评审确认、上线确认。
- 通知失败原因、重试状态、测试结果和 MR 链接。

### 5.2 多维表格：研发流水线数据库

每条需求是一行，多维表格不是装饰，而是管理者真实可用的需求台账。

建议字段：

| 字段 | 含义 |
| --- | --- |
| 需求标题 | 用户输入或飞书文档标题 |
| 来源 | bot_message / feishu_doc / manual |
| 状态 | Pending / Running / WaitingApproval / Failed / Delivered |
| 当前阶段 | 当前 Stage 名称 |
| 发起人 | 飞书 open_id / user_id |
| 审批人 | Tech Lead / Reviewer |
| 目标仓库 | repo URL 或 repo key |
| 目标分支 | base branch |
| 风险等级 | low / medium / high |
| 测试结果 | passed / failed / skipped |
| MR 链接 | 自动回填 |
| Token 成本 | 自动统计 |
| 最近失败原因 | 自动回填 |

### 5.3 飞书文档：研发产物沉淀

自动生成并维护：

- 结构化需求文档
- 技术方案文档
- 测试报告
- AI Review 报告
- 交付总结

### 5.4 Web 控制台：深度调试和演示

Web 不是主入口，而是面向演示、调试和审计的控制台。

必须展示：

- Pipeline DAG / 阶段时间线
- 每个 Agent 的输入、输出、模型和 Token 成本
- Artifact 列表
- Diff Viewer
- 测试日志
- 审批历史
- 重试次数和失败原因

## 8. Stage 输入输出契约

每个 Stage 都必须产出结构化 Artifact，并写入数据库和飞书产物。

| Stage | 输入 | 输出 |
| --- | --- | --- |
| RequirementAnalysis | 用户消息、飞书文档、仓库信息 | `requirement.json`、结构化需求文档 |
| FeishuContextBuild | 飞书链接、群聊上下文、多维表格历史记录 | `context_pack.json` |
| SolutionDesign | 结构化需求、代码库目录、上下文包 | `solution.json`、技术方案文档 |
| DesignApprovalCheckpoint | 技术方案 | 审批决策、Reject 理由 |
| CodeGeneration | 技术方案、目标仓库、相关文件 | patch / branch diff |
| TestGeneration | Diff、验收标准 | 测试文件、测试计划 |
| TestExecution | 测试计划、代码工作区 | `test_report.json`、日志 |
| AIReview | Diff、方案、测试结果 | `review_report.json`、风险列表 |
| HumanReviewCheckpoint | Diff、Review、测试报告 | 人工确认结果 |
| MRCreation | 已确认 Diff、交付摘要 | MR/PR 链接、变更摘要 |
| FeishuDeliverySummary | 全部 Artifact | 飞书交付总结、多维表格回填 |

## 9. Agent 设计

### 9.1 Requirement Agent

职责：

- 从自然语言、飞书文档、群聊片段中抽取需求。
- 识别目标用户、业务目标、范围、验收标准、开放问题。
- 判断需求是否足够进入研发，不足时生成澄清问题。

输出示例：

```json
{
  "title": "Pipeline 失败阶段自动重试",
  "background": "当前失败后只能人工重跑，现场演示和真实交付都缺少恢复能力。",
  "user_stories": [
    "作为研发负责人，我希望失败阶段能按配置自动重试，以减少偶发失败带来的人工干预。"
  ],
  "acceptance_criteria": [
    "每个 Stage 支持最大重试次数配置",
    "失败后自动重试直到成功或达到上限",
    "飞书卡片和 Web 控制台展示当前重试次数"
  ],
  "open_questions": [],
  "risk_level": "medium"
}
```

### 9.2 Feishu Context Agent

这是项目的特色 Agent。

职责：

- 读取飞书文档内容。
- 查询多维表格中的历史需求、状态和交付记录。
- 读取任务、人员、审批人等组织上下文。
- 将飞书上下文整理成可被其他 Agent 使用的上下文包。

实现建议：

- 稳定链路优先用飞书 OpenAPI 和 SDK。
- Agent 调试、运维和补充能力可通过 `lark-cli` 执行。
- `lark-cli` 默认使用 JSON 输出、dry-run 和最小权限配置，Tool Call 全部记录审计日志。

### 9.3 Solution Architect Agent

职责：

- 分析仓库目录和相关文件。
- 设计 API、数据模型和 UI 改动。
- 输出文件变更清单、测试计划、风险和回滚方案。

输出示例：

```json
{
  "summary": "为 Pipeline Stage 增加自动重试能力",
  "files_to_change": [
    "apps/api-go/internal/pipeline/executor.go",
    "apps/api-go/internal/pipeline/schema.go",
    "apps/web/src/pages/Monitoring.tsx"
  ],
  "api_design": [
    "Pipeline template 增加 max_retry_count",
    "Stage runtime 增加 retry_count 和 last_error"
  ],
  "test_plan": [
    "新增 executor 自动重试单元测试",
    "验证达到上限后进入 failed 状态"
  ],
  "rollback_plan": [
    "关闭模板 retry 配置即可恢复旧行为"
  ]
}
```

### 9.4 Code Agent

职责：

- 基于技术方案修改代码。
- 每轮只处理有限文件，降低误改风险。
- 输出 patch、变更摘要和潜在风险。
- 不能直接写主分支，必须在工作区或临时分支中执行。

### 9.5 Test Agent

职责：

- 生成或补充单元测试、集成测试。
- 运行项目已有测试命令。
- 解释失败原因，区分代码缺陷、环境缺失和测试不稳定。

### 9.6 Review Agent

建议拆成多个子 Reviewer，以体现多 Agent 协作：

- Correctness Reviewer：正确性和边界条件。
- Security Reviewer：凭证、权限、注入、敏感信息。
- Maintainability Reviewer：结构、重复、复杂度。
- Feishu API Permission Reviewer：飞书权限范围和 API 调用风险。
- UX Reviewer：审批卡片、控制台和用户路径是否清晰。

### 9.7 Release Agent

职责：

- 创建分支。
- 提交 commit。
- 创建 MR/PR。
- 生成 MR 描述和行级 Diff 摘要。
- 回写飞书多维表格。
- 在飞书群发送交付总结。

## 10. Human-in-the-Loop 设计

至少保留两个强制检查点，建议 MVP 做两个，演示版可做三个。

### 检查点 1：技术方案审批

触发时机：`SolutionDesign` 完成后。

飞书卡片展示：

- 需求摘要
- 影响范围
- 文件变更清单
- API / 数据模型设计
- 风险等级
- 测试计划
- 回滚方案

操作：

- `Approve`：继续进入代码生成。
- `Reject`：填写拒绝原因，回退到 `SolutionDesign` 重做。
- `Request Changes`：带修改建议重做方案。

### 检查点 2：代码评审确认

触发时机：`AIReview` 和测试执行完成后。

飞书卡片展示：

- Diff 摘要
- 测试结果
- AI Review 问题列表
- 安全风险
- MR 草稿描述

操作：

- `Approve`：进入 MR 创建和交付总结。
- `Reject`：携带原因回退到 `CodeGeneration` 或 `TestGeneration`。

### 检查点 3：上线前确认（演示增强项）

触发时机：MR 创建前或创建后。

展示：

- MR 链接
- 回滚方案
- 变更摘要
- 飞书通知文案

## 11. 飞书 API 和 lark-cli 融合点

| 飞书能力 | 项目用途 | MVP 优先级 |
| --- | --- | --- |
| 机器人消息 | 需求入口、状态通知、交付通知 | P0 |
| 消息卡片 | 审批、Reject 原因、阶段状态 | P0 |
| 云文档 | PRD 读取、方案文档、交付总结 | P0 |
| 多维表格 | 需求台账、状态看板、指标沉淀 | P0 |
| 通讯录 | 识别发起人、审批人、负责人 | P1 |
| 任务 | 自动创建研发任务 | P1 |
| 日历 | 安排评审会议 | P2 |
| lark-cli | Agent 工具层、调试、dry-run、自省 | P1 |

实现原则：

- 后端业务链路使用 OpenAPI / SDK，保证稳定和可控。
- lark-cli 用作 Agent Tool Adapter，适合执行飞书查询、文档读写、调试命令和结构化输出。
- 所有飞书写操作必须落审计日志。
- 对外部输入做 Prompt Injection 检测，特别是飞书文档和群聊内容。
- 飞书 API 权限按最小权限申请，演示环境用独立应用和测试群。

## 13. API 设计

必须坚持 API-First，Web、飞书卡片回调和 Agent 工具都调用同一套核心 API。

```text
GET    /api/pipeline-templates
POST   /api/pipeline-runs
GET    /api/pipeline-runs
GET    /api/pipeline-runs/{id}
POST   /api/pipeline-runs/{id}/start
POST   /api/pipeline-runs/{id}/pause
POST   /api/pipeline-runs/{id}/resume
POST   /api/pipeline-runs/{id}/terminate

GET    /api/pipeline-runs/{id}/stages
GET    /api/pipeline-runs/{id}/artifacts
GET    /api/pipeline-runs/{id}/agent-runs
GET    /api/pipeline-runs/{id}/checkpoints
GET    /api/pipeline-runs/{id}/code-diff

POST   /api/checkpoints/{checkpointID}/approve
POST   /api/checkpoints/{checkpointID}/reject

POST   /api/pipeline-runs/{id}/execute-changes
GET    /api/pipeline-runs/{id}/deliveries
GET    /api/git-deliveries/{deliveryID}

POST   /public/feishu/events
POST   /public/feishu/card/callback
```

需要补齐：

- OpenAPI/Swagger 文档与实际路由保持同步。
- `Reject` 请求体必须包含 `reason` 和可选 `target_stage_key`。
- 所有状态变更 API 要写入 Audit Log。

## 14. MVP 交付范围

### P0：参赛最小可赢版本

- 飞书机器人接收需求并创建 Pipeline Run。
- 支持从飞书文档读取 PRD。

## 15. 安全与权限

这是答辩重点，必须显式设计。

- Agent 不能直接写主分支，只能写工作区或临时分支。
- 高风险操作必须经过人工审批。
- 飞书 API 使用最小权限，演示应用和真实应用隔离。
- 所有 Tool Call、Prompt、输出、审批决策都记录审计日志。
- lark-cli 写操作默认 dry-run，审批后才执行真实写入。
- 对飞书文档和群消息中的外部内容做 Prompt Injection 检测。
- 敏感信息脱敏后再进入 LLM 上下文。
- MR 创建前必须展示 Diff、测试结果和风险摘要。

## 16. 可观测性

不要只做进度条。控制台和飞书总结应展示：

- 每个阶段耗时。
- 每个 Agent Token 消耗。
- 阶段失败次数和重试次数。
- Reject 原因统计。
- AI 修改了哪些文件。
- 测试通过率。
- 从需求创建到 MR 创建的总耗时。

这些指标能让项目从比赛 Demo 变成团队可运营的交付系统。

## 17. 端到端演示脚本

推荐演示需求：

```text
为 Pipeline 增加失败阶段自动重试能力，支持最大重试次数配置，并在飞书消息卡片和 Web 控制台中展示当前重试次数。
```

演示步骤：

1. 在飞书文档写需求和验收标准。
2. 在研发群 @DevFlow Bot 发起需求。
3. 展示多维表格新增需求记录。
4. 展示 Pipeline 自动进入需求分析和方案设计。
5. 展示飞书卡片中的方案审批，先 Reject 一次并填写原因。
6. 展示 Agent 携带 Reject 理由重做方案。
7. Approve 后进入代码生成和测试。
8. 展示 Web 控制台 Diff、测试日志、AI Review。
9. 在飞书卡片中确认代码评审。
10. 自动创建 MR/PR。
11. 飞书群收到交付总结，多维表格回填 MR 链接和测试结果。

这个演示能同时覆盖赛题要求的 Pipeline、Agent 编排、Human-in-the-Loop、API-First、端到端交付和自举能力。

## 18. 仓库改造路线

### 第一阶段：收敛产品主线

- 将 README 和文档统一为「飞书原生 AI 需求到 MR 交付平台」。
- 明确 Web 控制台不是官网，而是 Pipeline 驾驶舱。
- 清理或降级与主线无关的页面和功能描述。

### 第二阶段：补齐 Pipeline Core

- 固化 StageDefinition、StageRun、Artifact、Checkpoint、AgentRun 数据模型。
- 实现 Reject 回退到指定阶段。
- 实现 Stage 自动重试和最大次数配置。
- 将每个 Stage 的输入输出落库。

### 第三阶段：打通飞书闭环

- Bot 消息创建 Pipeline Run。
- 飞书文档读取 PRD。
- 多维表格创建和更新记录。
- 消息卡片支持 Approve/Reject。
- 生成方案文档和交付总结文档。

### 第四阶段：打通工程交付

- Workspace Manager 拉取或定位目标仓库。
- Code Agent 产出 patch。
- Test Runner 执行测试。
- Diff Viewer 展示变更。
- Git Provider 创建分支、提交、MR/PR。

### 第五阶段：完善演示和答辩材料

- 准备固定演示需求。
- 准备现场命题应急模板。
- 补 Docker Compose、安装指南、测试报告、OpenAPI 文档。
- 录制或截图关键链路，防止现场网络或飞书权限异常。

## 19. 最终价值表达

答辩时建议这样表达：

**我们不是做一个 AI 写代码工具，而是做一个飞书原生的 AI 研发交付系统。飞书负责协作入口和组织上下文，DevFlow Engine 负责编排，Agent 负责执行，人类负责关键决策。最终让研发流程从“人推动工具”变成“AI 推动流程，人审批关键节点”。**

项目成功的判断标准不是接入了多少飞书 API，而是：

- 需求是否能从飞书自然进入研发流程。
- 每个阶段是否有可审计产物。
- 人类是否能在关键节点做有效决策。
- AI 是否能对真实仓库产生可验证代码变更。
- 最终交付是否能沉淀回飞书，成为团队资产。
